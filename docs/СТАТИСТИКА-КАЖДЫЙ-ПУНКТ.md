# Статистика: откуда каждая цифра и каждая копейка

Документ описывает **каждый показатель** статистики: где считается, из каких таблиц/колонок, по каким условиям и какая запись в БД даёт вклад в сумму.

**Файлы бэкенда:**
- Агрегат (все парки): `backend/routes/admin-stats.js` — `GET /statistics/aggregate`
- По парку: `backend/routes/admin-stats.js` — `GET /parks/:parkId/statistics`
- Менеджер: `backend/routes/manager.js` — `GET /statistics` (парк из токена, логика та же что по парку)

**Даты везде по МСК.** Фильтры строятся в `backend/utils/moscow-time.js`: `getMoscowDateFilter(column, date)` и `getMoscowPeriodFilter(column, dateStart, dateEnd)`.

## 1. Финансы

### 1.1 topupsReal
- **Где:** `admin-stats.js` (aggregate ~41–42, по парку ~174–175).
- **Таблица:** `balance_history` (alias `bh`), связь `users` → `drivers`.
- **Условие:** `bh.type = 'topup'` и `(bh.amountType = 'real' OR bh.amountType IS NULL)`.
- **Формула:** `SUM(bh.amount)` по отфильтрованным строкам. Период — `bh.createdAt` в выбранном диапазоне по МСК; по парку ещё `d.parkId = ?`.
- **Откуда копейки:** Любая запись в `balance_history` с `type='topup'` и `amountType='real'` (или NULL). Пишутся при пополнении (реальные деньги), см. `utils/balance.js` — `addBalance`.

### 1.2 topupsRealCount
- **Где:** те же запросы.
- **Формула:** `COUNT(CASE WHEN bh.type = 'topup' AND (amountType = 'real' OR amountType IS NULL) THEN 1 END)` — число таких записей.

### 1.3 topupsUnreal
- **Условие:** `bh.type = 'topup'` и `bh.amountType = 'unreal'`.
- **Формула:** `SUM(bh.amount)`.
- **Копейки:** Записи `balance_history` с `type='topup'`, `amountType='unreal'` (бонусные пополнения).

### 1.4 topupsUnrealCount
- **Формула:** `COUNT(CASE WHEN bh.type = 'topup' AND bh.amountType = 'unreal' THEN 1 END)`.

### 1.5 spentReal
- **Где:** те же finance-запросы.
- **Условие:** `bh.type IN ('expense','waybill_fee')` и `(bh.amountType = 'real' OR bh.amountType IS NULL)`.
- **Формула:** `SUM(ABS(bh.amount))`.
- **Копейки:** Все списания реальных денег: ЭПЛ (`waybill_fee`), фотоконтроль, магазин игры, автозакрытие, удвоение очков и любые другие `expense` с `amountType='real'` или NULL.

### 1.6 spentUnreal
- **Условие:** `bh.type IN ('expense','waybill_fee')` и `bh.amountType = 'unreal'`.
- **Формула:** `SUM(ABS(bh.amount))`.
- **Копейки:** Те же типы операций, но списанные с бонусного баланса (`amountType='unreal'`).

### 1.7 spent
- **Формула:** `SUM(ABS(bh.amount))` по всем `bh.type IN ('expense','waybill_fee')` без разбивки по amountType. То есть **spent = spentReal + spentUnreal**.

### 1.8 systemBalanceReal
- **Где:** подзапрос в том же finance-запросе.
- **Таблица:** `users` (u2), `drivers` (d2). **Не** `balance_history`.
- **Формула:** `SUM(u2.balanceReal)` по пользователям, у которых есть запись в `drivers` (по парку — `d2.parkId = ?`). Мгновенный срез текущих балансов на момент запроса.

### 1.9 systemBalanceUnreal
- **Формула:** `SUM(u2.balanceUnreal)` по тем же пользователям (водителям). Тоже текущий срез из `users`.

## 2. ЭПЛ (путевые листы)

### 2.1 eplCount
- **Где:** `admin-stats.js` — подзапрос в operations (aggregate ~68, по парку ~206).
- **Таблица:** `epl`, связь через `drivers` (d) — `epl.driverId = d.id AND epl.parkId = d.parkId`. По парку: `d.parkId = ?`.
- **Поле даты:** `epl.createdAt` (тот же период по МСК, что и для финансов, но подставлен в отдельный фильтр `eplDateFilter`).
- **Формула:** `COUNT(DISTINCT epl.id)` — число уникальных ЭПЛ за период. Одна строка `epl` = один путевой.

### 2.2 eplAmount
- **Таблица:** `balance_history` + `users` + `drivers`.
- **Условие:** `bh.type = 'waybill_fee'`. Период — `bh.createdAt` по МСК; по парку — `d.parkId = ?`.
- **Формула:** `SUM(ABS(bh.amount))` по таким строкам.
- **Копейки:** Каждая запись в `balance_history` с `type='waybill_fee'` попадает в эту сумму. Такие записи создаёт только **создание ЭПЛ** (списание за путевой): сейчас только в `backend/services/fast-epl-pdf.js` при отрисовке PDF — `deductBalance(..., 'Списание за создание ЭПЛ', eplId, 'waybill_fee', ...)`. Сумма берётся из `waybill_rates.eplCreationFee` (или 25₽ по умолчанию). Одна запись на один ЭПЛ (проверка по `relatedEplId`).

### 2.3 eplAmountReal
- **Условие:** те же строки, что eplAmount, но `bh.amountType = 'real' OR bh.amountType IS NULL`.
- **Формула:** `SUM(ABS(bh.amount))`. Это часть eplAmount, списанная с реального баланса.

### 2.4 eplAmountUnreal
- **Условие:** `bh.type = 'waybill_fee'` и `bh.amountType = 'unreal'`.
- **Формула:** `SUM(ABS(bh.amount))`. Часть eplAmount с бонусного баланса. **eplAmount = eplAmountReal + eplAmountUnreal**.

## 3. Смены (закрытые / автозакрытые)

### 3.1 closedShiftsCount
- **Где:** подзапрос в operations.
- **Таблица:** `shifts` (s), связь через `drivers` (d): `s.driverId = d.userId AND s.parkId = d.parkId`, по парку `d.parkId = ?`.
- **Условие:** `s.status IN ('closed','auto_closed')`. Дата — по **моменту закрытия**: `COALESCE(s.closedAt, s.autoClosedAt)` в периоде по МСК (`sDateFilter`).
- **Формула:** `COUNT(DISTINCT s.id)` — число смен, закрытых (вручную или автоматически) в выбранном периоде.

### 3.2 autoClosedShiftsCount
- **Условие:** то же, но только `s.status = 'auto_closed'` и дата по `COALESCE(s.closedAt, s.autoClosedAt)`.
- **Формула:** `COUNT(DISTINCT s.id)` — подмножество закрытых смен, закрытых именно автоматически.

### 3.3 autoCloseAmount (и closedShiftsAmount)
- **Где:** после отдельного запроса `autoCloseQuery` (aggregate ~81–94, по парку ~231–248).
- **Таблица:** `balance_history` + `users` + `drivers`.
- **Условие:** `bh.type = 'expense'` и `bh.description LIKE '%Автозакрытие%'`. Период — `bh.createdAt` по МСК; по парку — `d.parkId = ?`.
- **Формула:**  
  - `autoCloseReal` = `SUM(ABS(bh.amount))` где ещё `(amountType = 'real' OR amountType IS NULL)`.  
  - `autoCloseUnreal` = то же где `amountType = 'unreal'`.  
  - **autoCloseAmount** = autoCloseReal + autoCloseUnreal (код: строки 94, 281).  
  - **closedShiftsAmount** в ответе просто равен autoCloseAmount (строки 101, 284).
- **Копейки:** Записи в `balance_history` с `type='expense'` и описанием, содержащим «Автозакрытие». Такие пишет `backend/services/epl-expiry-job.js` при автозакрытии смены: `deductBalance(..., feeDesc, ..., 'expense')`, где `feeDesc = 'Автозакрытие смены. С баланса списано X р.'`. Сумма — из `waybill_rates.autoCloseFee` (или настройки парка).

## 4. Фотоконтроль

### 4.1 photoControlCount
- **Где:** подзапрос в operations.
- **Таблица:** `photo_control_applications` (pca). По парку: `pca.parkId = ?`.
- **Поле даты:** `pca.createdAt` в периоде по МСК (`fcDateFilter`).
- **Формула:** `COUNT(*)` — число заявок на фотоконтроль (любой статус), созданных в периоде. Не фильтруется по status.

### 4.2 photoControlAmount
- **Таблица:** `balance_history` + `users` + `drivers`.
- **Условие:** `bh.type = 'expense'` и `bh.description LIKE '%Фотоконтроль%'`. Период — `bh.createdAt`; по парку — `d.parkId = ?`.
- **Формула:** `SUM(ABS(bh.amount))`.
- **Копейки:** Записи создаются при оплате фотоконтроля водителем: `backend/routes/driver.js` — `deductBalance(..., 'Фотоконтроль', null, 'expense', ...)`. Сумма — из `park_photo_control_settings.price` (по умолчанию 150₽).

## 5. Игра (магазин и удвоение очков)

Все поля — только в статистике **по парку** (в aggregate по играм только общий gameSpent).

### 5.1 gameSpent (в aggregate)
- **Условие:** `bh.type = 'expense'` и `bh.description LIKE '%Магазин игры%'`.
- **Формула:** `SUM(ABS(bh.amount))` без разбивки по amountType.

### 5.2 gameSpentReal (по парку)
- **Условие:** то же описание `LIKE '%Магазин игры%'` и `(bh.amountType = 'real' OR bh.amountType IS NULL)`.
- **Формула:** `SUM(ABS(bh.amount))`.

### 5.3 gameSpentUnreal (по парку)
- **Условие:** `LIKE '%Магазин игры%'` и `bh.amountType = 'unreal'`.
- **Формула:** `SUM(ABS(bh.amount))`. **gameSpent = gameSpentReal + gameSpentUnreal** (строка 282).

### 5.4 gameSpentMagnet
- **Условие:** `bh.type = 'expense'` и `bh.description LIKE '%Магазин игры: magnet%'`.
- **Копейки:** Покупки предмета magnet в игре. Запись: `driver.js` — `description = 'Магазин игры: magnet'` (или с суффиксом `×N`).

### 5.5 gameSpentNitro
- **Условие:** `bh.description LIKE '%Магазин игры: nitro%'`.

### 5.6 gameSpentJump
- **Условие:** `bh.description LIKE '%Магазин игры: jump%'`.

### 5.7 gameSpentExtraLife
- **Условие:** `bh.description LIKE '%Магазин игры: extra_life%'`.

### 5.8 doubleCoinsSpent
- **Условие:** `bh.type = 'expense'` и `bh.description LIKE '%Удвоение очков в игре%'`.
- **Копейки:** Запись создаётся в `driver.js`: `deductBalance(..., 'Удвоение очков в игре', null, 'expense', ...)` (цена DOUBLE_COINS_PRICE).

## 6. Прочее

### 6.1 newDriversCount (только по парку)
- **Где:** отдельный запрос с фильтрами по периоду (admin-stats.js ~257–278).
- **Таблица:** `drivers` (d), `d.parkId = ?`.
- **Поле даты:** `d.createdAt` в выбранном периоде по МСК (`newDriversFilter`).
- **Формула:** `COUNT(DISTINCT CASE WHEN <newDriversFilter> THEN d.id END)` — число водителей, созданных в парке за период.

### 6.2 newBindingsCount (только по парку)
- **Поле даты:** `d.updatedAt` в периоде по МСК (`newBindingsFilter`).
- **Условие:** у водителя есть привязка `d.carId IS NOT NULL` и обновление в период.
- **Формула:** `COUNT(DISTINCT CASE WHEN d.carId IS NOT NULL AND <newBindingsFilter> THEN d.carId END)` — число машин, которые получили привязку к водителю в период (по факту обновления записи водителя).

### 6.3 forecast
- **Где:** считается в коде после получения financeStats (aggregate ~104, по парку ~285).
- **Формула:** `Math.floor(systemBalanceReal / Math.max(spent / 30, 1))`. То есть: текущий суммарный реальный баланс водителей парка делим на средние дневные траты за период (spent / 30), минимум 1 в знаменателе. Получается ориентировочное «на сколько дней хватит баланса при текущем расходе». Не из БД напрямую — только из уже посчитанных systemBalanceReal и spent.

---

## 7. Базовая сводка (users, cars, bindings)

- **users:** `COUNT(DISTINCT d.id)` по `drivers` (d). По парку: `d.parkId = ?`. Это количество водителей.
- **cars:** `COUNT(DISTINCT c.id)` по `cars` (c). По парку: `c.parkId = ?`.
- **bindings:** `COUNT(DISTINCT d.id)` по `drivers` где `d.carId IS NOT NULL`. По парку: `d.parkId = ?`. Число водителей с привязанной машиной.

## 8. Где создаются записи balance_history (тип и описание)

| type          | description (пример)                    | Где создаётся                          |
|---------------|------------------------------------------|----------------------------------------|
| topup         | —                                       | Пополнение (addBalance)                |
| waybill_fee   | Списание за создание ЭПЛ                | fast-epl-pdf.js при создании ЭПЛ       |
| expense       | Фотоконтроль                            | driver.js — оплата фотоконтроля        |
| expense       | Автозакрытие смены. С баланса списано X р. | epl-expiry-job.js при автозакрытии   |
| expense       | Магазин игры: magnet / nitro / jump / extra_life | driver.js — покупка в игре     |
| expense       | Удвоение очков в игре                   | driver.js — покупка удвоения очков     |

Для всех списаний `amount` в `balance_history` хранится **отрицательным**; в статистике везде используется `ABS(bh.amount)`.

---

Если нужно проследить конкретную сумму: откройте `balance_history` по нужному парку (через userId → drivers.parkId) и фильтруйте по `type`, `amountType` и `description` как в таблице выше; дата — `createdAt` в МСК.
