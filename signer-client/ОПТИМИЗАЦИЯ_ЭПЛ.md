# Оптимизация скорости обработки ЭПЛ

## Текущая схема

1. **Signer (sign.js + taxcom-create.js):** создаёт и подписывает ЭПЛ (Т1–Т4), пробует забрать QR и PDF, шлёт epl-created на сайт. **Браузер переиспользуется между ЭПЛ** — выход, логин диспетчера, следующий путевой без перезапуска.
2. **QR-fetcher (qr-fetcher.js):** берёт ЭПЛ без QR (≥2 мин), заходит в Такском, снимает QR, шлёт на сайт. Браузер остаётся открытым.

## Что сделано для ускорения

### 1. Убрано ожидание 120 сек в signer
- **Было:** после подписания Т4 — ждём 2 мин, пока QR появится в ГИС, потом забираем.
- **Стало:** `TAXCOM_QR_WAIT_SECONDS=0` — сразу отправляем epl-created (mintransId + PDF), QR забирает qr-fetcher.
- **Выигрыш:** ~2 мин на каждый ЭПЛ.

### 2. Signer переиспользует браузер между ЭПЛ
- **Было:** после каждого ЭПЛ — закрыть браузер, следующий ЭПЛ — запуск, логин, форма.
- **Стало:** контекст и страница передаются в следующий вызов; если браузер ещё открыт — logout → login диспетчера → сразу «Создать путевой». При ошибке — сброс, следующий ЭПЛ запустит браузер заново.
- **Выигрыш:** ~15–30 сек на каждый ЭПЛ (без запуска Chromium и загрузки профиля).

### 3. QR-fetcher держит браузер открытым
- **Было:** при пустой очереди — закрыть браузер, пауза 25 сек, новый запуск, логин, опрос.
- **Стало:** браузер остаётся открытым (`QR_FETCH_KEEP_BROWSER=1` по умолчанию), при пустой очереди — пауза 15 сек, повтор.
- **Выигрыш:** ~10–15 сек на каждый цикл.

### 4. QR-fetcher: retry при отсутствии QR
- Если QR не найден — обновление страницы и повторная попытка, затем пропуск при неудаче.
- **Выигрыш:** меньше ложных пропусков из-за задержек загрузки.

### 5. Логирование
- Signer: время обработки ЭПЛ (в секундах), пометка «[переиспользую браузер]».
- QR-fetcher: время получения QR, пометка «Переиспользую браузер» / «Запуск браузера».

### 6. Пауза при пустой очереди QR
- Было: 25 сек. Стало: 15 сек.

### 7. Рекомендуемые настройки в .env

```
TAXCOM_QR_WAIT_SECONDS=0
QR_FETCH_KEEP_BROWSER=1
QR_FETCH_IDLE_SLEEP_SEC=15
QR_FETCH_MIN_AGE_MINUTES=2
```

## Оценка времени на 1 ЭПЛ

| Этап                         | Было        | Стало        |
|-----------------------------|-------------|-------------|
| Запуск браузера + логин     | ~15–30 сек  | 0 (переиспользование) |
| Заполнение + подпись Т1–Т4  | ~2–3 мин    | ~2–3 мин     |
| Ожидание QR (если не готов) | до 2 мин    | 0            |
| Сбор PDF                    | ~2–5 сек    | ~2–5 сек     |
| QR забирает qr-fetcher      | ~2 мин после подписания | так же  |

**Итог:** на 1 ЭПЛ экономим до 2+ минут; signer освобождается быстрее; браузер не перезапускается между заявками.

## Потоковая работа

1. Signer подписывает ЭПЛ 1 → epl-created (без QR) → **без закрытия браузера** сразу берёт ЭПЛ 2 (logout → login → создание).
2. QR-fetcher через ~2 мин забирает QR для ЭПЛ 1 и шлёт на сайт.
3. Параллельно signer уже работает с ЭПЛ 2 и дальше, используя тот же браузер.

Окна signer и qr-fetcher — разные процессы в одном `sign.js`; qr-fetcher опрашивает API в фоне и держит свой браузер открытым.

## Важно

- **TAXCOM_KEEP_BROWSER_OPEN** — больше не влияет на signer: браузер по умолчанию переиспользуется, а не закрывается. Сброс context/page происходит только при ошибке.
- Браузер qr-fetcher — отдельный; его можно держать открытым и переиспользовать.

## Режим трёх воркеров (диспетчер / медик / механик)

Заявки можно обрабатывать конвейером: три отдельных процесса с разными логинами и окнами браузера.

1. **dispatcher-worker.js** — только создание ЭПЛ и подпись Т1. Опрашивает pending-creation, берёт заявки без mintransId. После Т1 шлёт epl-created (mintransId), заявка переходит к медику.
2. **medic-worker.js** — только подпись Т2. Берёт заявки с mintransId и titulStatus.t1=signed, titulStatus.t2 не signed. Шлёт titul-progress по Т2.
3. **mechanic-worker.js** — только подпись Т3 и Т4. Берёт заявки с подписанными Т1 и Т2 (из pending-creation). Шлёт titul-progress по Т3/Т4. В логе: «Заявка для Т3/Т4: WB-…».
4. **qr-fetcher.js** — запрашивает **батч** ЭПЛ без QR (до 5 по умолчанию, `QR_FETCH_BATCH_SIZE`), только с подписанными Т1–Т4 (QR в Такском появляется после Т4). Обрабатывает подряд без повторных запросов API. Опционально `QR_FETCH_MIN_AGE_MINUTES=0` — в очередь попадают и только что подписанные ЭПЛ.

Запуск — два варианта:

**1) Один скрипт (все 4 воркера в одном окне, логи с префиксами [Dispatcher], [Medic], [Mechanic], [QR]):**

Скрипт лежит в папке **signer-client**. В командной строке (cmd) введи по очереди — каждую строку отдельно, Enter в конце:

```
cd signer-client
node run-workers.js
```

Если ты уже в `signer-client`, достаточно одной команды:

```
node run-workers.js
```

Или из корня проекта (папка «Сайт Такси1») одной строкой:

```
node signer-client\run-workers.js
```

Остановка: Ctrl+C (все процессы завершатся).

**2) Четыре отдельные консоли (удобно для отладки по ролям):**

```bash
node dispatcher-worker.js
node medic-worker.js
node mechanic-worker.js
node qr-fetcher.js
```

Логины диспетчера/медика/механика берутся из .env (TAKSKOM_DISPATCHER_PHONE, TAKSKOM_MEDIC_PHONE, TAKSKOM_MECHANIC_PHONE и пароли).

Конвейер работает **отдельно от бэкенда**: бэкенд и фронт (в т.ч. на Netlify) могут быть развёрнуты где угодно; run-workers.js (или четыре воркера) крутится на ПК клиники или на сервере с Node и настроенным .env. API_URL в .env должен указывать на адрес бэкенда.

**Важно:** при запуске через `node signer-client\run-workers.js` диспетчер, медик и механик получают **отдельные** профили: `signer-client/profiles/dispatcher`, `profiles/medic`, `profiles/mechanic`. При **первом** запуске скрипт копирует твой профиль с КриптоПро (из папки User Data рядом с CHROMIUM_GOST_PATH или из CRYPTOPRO_SOURCE_PROFILE в .env) в эти три папки — в каждом окне будет КриптоПро и подпись. **Перед первым запуском закрой браузер Chromium**, иначе копирование может не удасться. После копирования откроются три окна, в каждом войди в Такском под своим логином (диспетчер, медик, механик).

Если запускаешь воркеры по отдельности (не через run-workers.js), задай в .env разные профили вручную:

- DISPATCHER_USER_DATA_DIR
- MEDIC_USER_DATA_DIR
- MECHANIC_USER_DATA_DIR
